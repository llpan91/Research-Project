## Q1. UMI 夹爪上的二维码（Fiducial Markers）如何工作？

### 1.1 是什么

UMI 夹爪手指上贴有 **Fiducial Markers（基准标记）**，通常为 ArUco 类型的黑白方块图案。这些标记由腕部安装的 GoPro 相机持续拍摄。

### 1.2 核心功能：连续追踪夹爪手指宽度

- 相机检测两个手指上标记的**相对位置**，实时计算 **gripper width（手指张开宽度）**
- 实现**连续**的夹爪宽度记录（而非传统的二值开/合）
- 追踪频率与视频帧率同步

### 1.3 为什么需要连续追踪

| 对比项 | 二值控制（开/关） | 连续控制（UMI） |
|--------|------------------|----------------|
| 精度 | 只有两个状态 | 任意宽度值 |
| 适用任务 | 简单抓放 | 抛掷、精确释放 |
| 力控制 | 无法控制 | 通过软手指变形隐式控制夹持力 |

### 1.4 额外作用

- **标定辅助**：在 HD6 运动学过滤中，也使用桌面上放置的 fiducial markers 来标定 SLAM 坐标系与机器人基座坐标系之间的变换关系
- **跨机器人部署**：通过标定同一组 fiducial markers，可以将 SLAM 输出的绝对位姿转换为机器人基座坐标系下的绝对动作

## Q2. "基于运动学的数据过滤机制"（HD6）是做什么？

### 2.1 核心思想

UMI 的数据采集过程是**机器人无关的**（robot-agnostic）——任何人都可以手持夹爪在任意环境中采集数据。但部署时需要在**特定机器人**上执行，因此需要过滤掉该机器人无法执行的轨迹。

### 2.2 具体流程

```
采集的全部轨迹（机器人无关）
        │
        ▼
  已知目标机器人的：
  - 基座位置（base location）
  - 运动学模型（kinematics）
        │
        ▼
  SLAM 恢复的绝对末端位姿
        │
        ▼
  逐条轨迹检查：
  ├─ 关节角度是否超出限位？ → 超出则过滤
  ├─ 末端位置是否在工作空间内？ → 超出则过滤
  ├─ 运动速度/加速度是否超限？ → 超出则过滤
  └─ 通过所有检查 → 保留
        │
        ▼
  过滤后的数据集（适配目标机器人）
```

### 2.3 意义

- **一次采集，多机器人复用**：同一批示教数据可以分别为 UR5（6DoF）和 Franka（7DoF）过滤出各自可行的子集
- **确保策略可行性**：训练后的策略只会输出目标机器人物理上可执行的动作
- **无需重新采集**：换机器人时只需重新过滤，不需要重新示教

### 2.4 论文实验验证

在 Cup Arrangement 任务中，UMI 将同一套数据分别部署到 UR5 和 Franka FR2 上，Franka 达到 90% 成功率（18/20），2 次失败均为关节限位问题（可通过调整机器人安装位置避免）。

## Q3. UMI 中涉及哪些 Diffusion Policy 建模？

### 3.1 为什么需要 Diffusion Policy

人类示教数据天然具有**多模态性**（multimodality）：对于同一个观测，可能存在多种合理的动作。例如 Cup Arrangement 任务中，杯子把手正对机器人时，可以选择顺时针或逆时针旋转——这形成了多模态动作分布。传统的回归损失（如 MSE）会将多种模式平均，导致策略输出无效的"折中"动作。

### 3.2 Diffusion Policy 在 UMI 中的具体角色

Diffusion Policy 是 UMI 的**核心策略网络**，负责以下建模：

| 建模内容 | 说明 |
|---------|------|
| **输入（观测）** | 同步的 RGB 图像序列、相对 EE 位姿序列、夹爪宽度序列 |
| **输出（动作）** | 未来一段时间的**相对 EE 位姿轨迹** + **夹爪宽度轨迹** |
| **多模态分布** | 通过去噪扩散过程建模，能够从噪声中采样出不同模式的动作 |

### 3.3 具体配置

- **动作表示**：相对轨迹（Relative Trajectory），即所有未来位姿相对于当前 EE 位姿的 SE(3) 变换
- **观测历史**：observation horizon = 2 步（提供速度信息）
- **视觉编码器**：
  - 简单任务（Cup Arrangement, Tossing, Cloth Folding）：ResNet-34
  - 复杂长时域任务（Dish Washing）：CLIP 预训练的 ViT-B/16（进一步微调）
  - 开放环境泛化实验：CLIP 预训练的 ViT-L/14
- **镜像图像处理**：侧面镜子中的图像经过数字反射（digital reflection）后裁剪出来，作为额外视觉输入

### 3.4 与其他框架的关系

论文指出 ACT（Action Chunking with Transformers）等其他框架也可以作为替代方案（drop-in replacement），Diffusion Policy 并非 UMI 唯一可选的策略架构。

## Q4. 侧面镜子的隐式立体视觉，具体需要什么预处理？

### 4.1 硬件设计

在 UMI 夹爪的 155° 鱼眼相机两侧，各放置一面**物理镜片**。由于鱼眼镜头视野足够宽，镜片中的反射图像也会被拍入同一帧画面中。

### 4.2 等价原理

每面镜子中的图像，等效于一个**虚拟相机**沿镜面对称位置拍摄的画面：

```
        [左虚拟相机]  ←  左镜面反射  ←  [主相机]  →  右镜面反射  →  [右虚拟相机]
```

- 三个视角（主 + 左镜 + 右镜）具有不同的光心位置，形成**立体视差**
- 这为策略网络提供了隐式的**深度感知**能力

### 4.3 关键预处理步骤：数字反射（Digital Reflection）

论文发现**直接使用镜像图像反而降低性能**（从 90% 降到 85%）。原因是：镜子中物体的运动方向与主画面中相反，会混淆具有平移等变性的视觉编码器。

因此必须进行**数字反射**处理：

```
原始鱼眼图像
     │
     ├─ 主视图区域 → 直接裁剪 → 输入1
     │
     ├─ 左镜区域 → 裁剪 → 水平翻转（左右镜像） → 输入2
     │
     └─ 右镜区域 → 裁剪 → 水平翻转（左右镜像） → 输入3
```

经过数字反射后：
- 三个视图中物体的**运动方向一致**
- 杯子把手等物体的**朝向一致**（论文 Fig.4(c) 展示了反射前后杯子把手方向的统一）
- 成功率从 85%（无镜子）/ 85%（镜子不翻转）提升到 **100%**

### 4.4 与真正双目的区别

| 对比项 | 真正双目 | UMI 隐式立体 |
|--------|---------|-------------|
| 硬件 | 两个相机 + 标定 | 一个相机 + 两面镜子 |
| 深度图 | 可显式计算稠密深度 | 不显式计算深度，由网络隐式学习 |
| 基线 | 固定已知 | 由镜子位置决定，较小 |
| 成本 | 高 | 极低（两面小镜子） |

UMI 并不做显式的立体匹配或深度估计，而是将三个视图都输入策略网络，让网络**自己学习利用视差信息**来感知深度。

## Q5. SLAM 精度（位置 6.1mm / 旋转 3.5°）处于什么水平？

### 5.1 UMI 的 SLAM 精度数据

论文 Fig.12 报告的精度（以 MoCap 为 ground truth）：

| 指标 | 位置误差 | 旋转误差 |
|------|---------|---------|
| 单夹爪 ATE（绝对轨迹误差） | **6.1 mm** | **3.5°** |
| 双夹爪间 RPE（相对位姿误差） | 10.1 mm | 0.8° |

### 5.2 与消费级 VIO 硬件的对比

| 系统 | 位置精度 | 旋转精度 | 备注 |
|------|---------|---------|------|
| **UMI (GoPro + ORB-SLAM3)** | **6.1 mm** | **3.5°** | 后处理，非实时 |
| Apple ARKit (iPhone/iPad) | 1-10 mm | 0.5-1° | 实时，专用硬件 |
| Intel RealSense T265 | 5-10 mm | ~1° | 双目+IMU，已停产 |
| Meta Quest 3 VIO | ~5 mm | ~1° | 多相机+IMU |
| ORB-SLAM3 (标准单目+IMU) | 10-30 mm | 1-3° | 依赖场景纹理 |

### 5.3 分析

- **位置精度 6.1mm**：对于操作任务来说**相当不错**，已经在亚厘米级别，足够支撑精细操作（如杯子摆放）
- **旋转误差 3.5°**：确实**偏大**，主要原因：
  1. GoPro 并非为精密定位设计，IMU 品质为消费级
  2. 鱼眼镜头虽然增大了视野但引入了更大的畸变
  3. 手持操作中存在频繁的快速旋转，对 IMU 积分漂移更敏感
  4. 论文使用的是**离线 SLAM**（后处理），而非实时 VIO——后处理虽有全局优化（loop closure），但 GoPro 的 IMU 预积分质量仍是瓶颈

### 5.4 为什么 UMI 仍然能工作

尽管旋转误差偏大，UMI 通过以下设计缓解了影响：

1. **相对轨迹表示（PD2.1）**：动作定义为相对于当前位姿的变换，短时间内的相对误差远小于绝对误差
2. **Diffusion Policy 的鲁棒性**：策略网络在有噪声的数据上训练，天然具有一定的容错能力
3. **双夹爪相对误差仅 0.8°**：对于双臂协作最关键的是两个夹爪之间的**相对姿态**，这个精度很高

### 5.5 论文承认的局限

论文在 Limitations（第 VIII 节）中明确指出：SLAM 系统依赖场景中的充足纹理，在纯白墙壁等纹理缺乏的环境中可能失效。未来可以通过额外的第三人称相机或在夹爪上增加 fiducial markers 来补充。

## Q6. "推理时延迟匹配"（Inference-Time Latency Matching）的具体算法实现

### 6.1 问题背景

训练时，UMI 夹爪的所有数据流（视频、IMU、夹爪宽度）由 GoPro 同步录制，**延迟为零**。但部署到真实机器人时，各传感器和执行器的延迟各不相同：

```
训练时（UMI 夹爪）：     所有流完美同步，延迟 = 0
部署时（真实机器人）：   各流延迟不同 → 分布外输入 → 策略性能下降
```

### 6.2 观测延迟补偿（PD1.1）

**目标**：将所有观测流对齐到同一时刻，消除流间时间差。

**具体步骤**：

```
步骤1：离线测量各传感器流的延迟
  - RGB 图像（10-20Hz）→ 延迟最大，记为 latency_img
  - 夹爪宽度（30Hz）    → 中等延迟，记为 latency_grip
  - EE 位姿（125Hz）    → 最小延迟，记为 latency_arm

步骤2：以最高延迟流（通常是图像流）为基准
  - 基准时刻 t_obs = 图像的采集时间戳

步骤3：对其他流进行时间对齐
  - 将 RGB 降采样到目标频率（10-20Hz）
  - 以每帧图像的时间戳 t_obs 为锚点
  - 对夹爪宽度和 EE 位姿进行线性插值，取 t_obs 时刻的值

步骤4（双臂场景）：
  - 两个相机通过最近邻帧匹配进行软同步
  - 最大误差 ≤ 1/60 秒

结果：输入策略的所有观测流在时间上完美对齐
```

### 6.3 执行延迟补偿（PD1.2）

**目标**：确保机器人在策略期望的时刻到达期望位姿。

**具体步骤**：

```
步骤1：离线测量执行延迟
  - 机械臂执行延迟 = 100ms（从发送指令到实际到达）
  - 夹爪执行延迟 = 120ms

步骤2：策略输出的动作序列带有时间戳
  策略在 t_output 时刻输出：
  [pose(t1), pose(t2), pose(t3), ...]
  每个 pose 对应一个期望到达时刻

步骤3：提前发送指令
  - 对于期望在 t_desired 到达的位姿：
    实际发送时刻 = t_desired - execution_latency
  - 机械臂指令提前 100ms 发送
  - 夹爪指令提前 120ms 发送

步骤4：丢弃已过期的动作
  - 策略预测的动作序列起始于 t_obs（观测时刻）
  - 但从 t_obs 到 t_output 已经过了推理时间
  - 再加上执行延迟，前几步动作已经"过期"
  - 系统丢弃 t < t_act 的动作，只执行未来时刻的动作

  其中 t_act = t_output + execution_latency
```

### 6.4 实验验证

在 Dynamic Tossing 任务中，延迟匹配的效果最为显著：
- **有延迟匹配**：87.5% 成功率（105/120）
- **无延迟匹配**：57.5% 成功率（69/120）

无延迟匹配时，机械臂运动出现明显抖动（jitter），肘关节速度曲线偏离期望值，导致抛掷时机和力度不准确。

## Q7. 解决 Action Imprecision 问题

### 7.1 问题根源

现有手持采集设备大多依赖**单目 SfM（Structure from Motion）**来恢复末端执行器轨迹，存在三个固有缺陷：

| 问题 | 说明 | 影响 |
|------|------|------|
| **尺度模糊** (scale ambiguity) | 单目相机无法恢复真实物理尺度 | 轨迹的绝对尺寸未知 |
| **运动模糊** (motion blur) | 快速运动时图像模糊 | 特征追踪失败，轨迹断裂 |
| **纹理不足** (insufficient texture) | 场景特征点稀少 | 追踪不稳定，漂移严重 |

结果：恢复的动作轨迹不够精确，限制系统只能完成粗糙的抓取放置任务。

### 7.2 UMI 的解决方案：HD4（IMU 感知追踪）

UMI 使用 GoPro 内置 IMU + 基于 ORB-SLAM3 的**惯性-单目 SLAM 系统**，逐一解决上述问题：

#### 解决尺度模糊
```
单目 SfM：轨迹 = k × 真实轨迹（k 未知）
                    │
加入 IMU 加速度计 ──┘
                    │
                    ▼
IMU 提供绝对加速度（m/s²）
→ 通过视觉-惯性联合优化，恢复真实的米制尺度
→ 轨迹具有绝对物理尺寸
```

#### 解决运动模糊
```
快速运动 → 图像模糊 → 视觉特征追踪失败
                              │
但 IMU 仍在 125Hz 正常工作 ──┘
                              │
                              ▼
惯性预积分（IMU preintegration）在视觉失效期间
维持位姿估计，直到视觉恢复后再次联合优化
→ 短暂的视觉失效不会导致追踪丢失
```

#### 解决纹理不足
```
白墙、桌面等低纹理区域 → 视觉特征稀少
                              │
IMU 提供旋转和加速度约束 ──┘
                              │
                              ▼
惯性约束为优化提供强先验
→ 即使视觉特征极少，仍可维持合理的位姿估计
```

### 7.3 关键技术细节

- **传感器**：GoPro 内置 3 轴加速度计 + 3 轴陀螺仪，通过 GPMF 格式嵌入 MP4 文件
- **SLAM 框架**：ORB-SLAM3，支持单目惯性模式
- **处理方式**：**离线后处理**（非实时），可进行全局 BA（Bundle Adjustment）和回环检测
- **坐标系**：输出位姿相对于 mapping video 的第一帧，后续通过 fiducial markers 标定转换到机器人坐标系

### 7.4 量化结果

与 MoCap（动作捕捉系统）ground truth 对比：
- 位置 ATE：**6.1 mm**（亚厘米级，满足精细操作需求）
- 旋转 ATE：**3.5°**（可接受，通过相对轨迹表示进一步缓解）

### 7.5 对比先前方法

| 方法 | 尺度 | 鲁棒性 | 精度 | 成本 |
|------|------|--------|------|------|
| 单目 SfM | 无绝对尺度 | 差（怕模糊/无纹理） | 低 | 低 |
| RGB-D 融合 | 有 | 中等 | 中等 | 高（需深度相机） |
| 外部动捕 MoCap | 有 | 高 | 极高 | 极高，不便携 |
| **UMI (惯性-单目 SLAM)** | **有** | **高** | **高** | **低（仅 GoPro）** |
